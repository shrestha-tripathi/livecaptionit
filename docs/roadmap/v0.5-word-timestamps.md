# v0.5 — Word-Level Timestamps + Confidence + Transcript Editor

**Status:** Roadmap / spec draft (NOT implementation-ready — see Open Questions)
**Author:** session 2026-06-10
**Related:** `public/whisper-worker.js`, `src/lib/whisperClient.ts`, `src/lib/agreement.ts`, `src/lib/transcript.ts`, `src/lib/sessionStore.ts`, `src/components/CaptionApp.script.ts`
**Estimated work:** ~1200-1600 LOC across 8 commits, ~3 working days
**Risk:** Medium-high (worker API breaking change + persisted-data migration)

---

## Problem

LiveCaptionIt's worker → parent message shape is currently text-only — each `transcribe` call returns a flat `{ text, durationMs }` payload. Three v0.4.3 features couldn't ship because they all need word-level timing and/or per-word confidence:

| Bailed feature | Reason it couldn't ship in v0.4.3 |
|---|---|
| **Confidence-coloured live tail** (Phase 1.7) | No per-word logprob exposed |
| **Word-level timestamps in .vtt / .srt** (Phase 2.5) | Worker returns a single string, no per-token start/end |
| **Speaker-turn manual editing** (Phase 3.6) | Editor needs to anchor edits to specific words to preserve timing |

All three want the SAME worker change (`return_timestamps: "word"` + a richer `Word[]` payload). Shipping them one-by-one would mean three worker-API breaking changes in three weeks. Bundling them into v0.5 means one breaking change, one migration, three visible wins.

There's also a slower-cooking pain point that v0.5 should address as a bonus:

| Friction | Concrete example |
|---|---|
| Transcript downloads have segment-grained timing | A 14-word segment that took 8s gets exported as one .vtt cue spanning 8s. Real subtitle workflows want ~3-4s cues at sentence boundaries. |
| Old shared/exported transcripts won't have word timing | We need a clean migration path so v0.4.x users restoring backups still work. |
| Hallucination guard runs AFTER decode | With per-word confidence we can mute-then-drop low-confidence words DURING the live render — better UX than "appear then disappear when force-commit reclaims them." |

## Current architecture (what we'd be changing)

### Worker — `public/whisper-worker.js:178-211`

```js
const result = await asr(audio, {
  chunk_length_s: 30,
  stride_length_s: 5,
  return_timestamps: false,   // ← change to "word" in v0.5
  language: "english",
  task: "transcribe",
  num_beams: 1,
  temperature: 0,
  no_repeat_ngram_size: 3,
  initial_prompt: vocabularyPrompt || undefined,
});
const text = (result && result.text ? result.text : "").trim();
self.postMessage({ type: "result", text, id, durationMs });
```

When `return_timestamps: "word"`, transformers.js returns:

```js
{
  text: "hello world how are you",
  chunks: [
    { text: "hello",  timestamp: [0.0, 0.42] },
    { text: " world", timestamp: [0.42, 0.84] },
    { text: " how",   timestamp: [0.84, 1.1]  },
    { text: " are",   timestamp: [1.1,  1.32] },
    { text: " you",   timestamp: [1.32, 1.58] }
  ]
}
```

Per-word **confidence** is NOT in the default output but is available via the encoder logprobs — see Open Question #2 for the path.

### Parent proxy — `src/lib/whisperClient.ts:23,189-195`

```ts
transcribeWindow: (audio: Float32Array) => Promise<WindowResult>;
// ...
transcribeWindow(audio: Float32Array) {
  return new Promise<WindowResult>((resolve, reject) => {
    const id = ++nextId;
    pending.set(id, { resolve, reject });
    worker.postMessage({ type: "transcribe", audio, id }, [audio.buffer]);
  });
}
```

`WindowResult` is `{ text: string; durationMs: number }` today.

### Agreement — `src/lib/agreement.ts`

LocalAgreement-2 operates on word arrays (split from `text`). It already produces a `committed: string[]` + `liveLine: string[]` interface. Adding per-word timestamps means changing those to `committed: Word[]` + `liveLine: Word[]` and updating two render paths in `CaptionApp.script.ts` to read `.text` instead of treating each element as a bare string.

### Transcript segment — `src/lib/transcript.ts:23-28`

```ts
export interface TranscriptSegment {
  /** Words committed in this batch. */
  words: string[];
  /** Milliseconds since session start when the batch was committed. */
  tMs: number;
}
```

This shape is persisted to IndexedDB via `sessionStore.ts` AND serialized in shareLink URLs AND in JSON exports. **Changing it = a migration event.**

### sessionStore — `src/lib/sessionStore.ts:214-262`

```ts
export const EXPORT_VERSION = 1 as const;
// ...
if (obj.version !== EXPORT_VERSION) {
  throw new Error(`Unsupported export version: ${String(obj.version)} ...`);
}
```

The validator already throws on version mismatch — v0.5 must bump EXPORT_VERSION to 2 AND add an `if (bundle.version === 1) { migrate }` branch in `importSessions`.

## Design options

### Option A — Ship features piecemeal across v0.4.4 / v0.4.5 / v0.4.6

Keep the existing worker API. Add per-word data via a SECOND worker call that's only fired when a feature needs it (e.g. `transcribeWithTimestamps` for export-time, separate from the live-tail loop).

| Pros | Cons |
|---|---|
| No breaking change to live captioning path | Double decode = 2× compute when export needed |
| Each visible feature can land independently | Confidence-coloured live tail still impossible (would need it on every tick) |
| Easy rollback | Tech debt: two paths for "transcribe", easy to drift |

**Verdict:** Insufficient. Confidence-coloured live tail is the most-asked feature and it specifically needs per-word data on the streaming path. Double-decode tax on every export is also ugly.

### Option B — Single worker change, big-bang release (chosen)

Switch `return_timestamps: "word"` ONCE. Change worker message shape to carry `chunks: Word[]` everywhere. Migrate downstream consumers (agreement, transcript, sessionStore) in a coordinated v0.5 release.

| Pros | Cons |
|---|---|
| All three deferred features unlock together | Larger blast radius — single bug can regress all three |
| One persisted-data migration, not three | Migration risk for users with v0.4.x IndexedDB sessions |
| Each feature ships its own commit AFTER the worker change | Worker call returns slightly more bytes (~50% larger payload — still tiny per tick) |
| Eliminates the eventual double-decode tax | Some commits in the chain are pure plumbing (low visible value alone) |

**Verdict:** Pick this. The features are coupled at the data level; ship the coupling once.

### Option C — Add a parallel "rich mode" toggle (worker stays default-text, opt-in to word mode)

Run `return_timestamps: "word"` only when user opts in (via a "Power features" toggle) OR per session. Two paths coexist.

| Pros | Cons |
|---|---|
| Conservative users keep the old fast path | Two ways everything works — testing matrix 2× |
| No persisted-data migration if opt-in is off-by-default | Users who turn it on AFTER recording need a backfill — same migration anyway |
| Lets us A/B perf | Doesn't actually save much: per-word adds <5% to decode time on whisper-base |

**Verdict:** Reject. The 5% extra cost is real but trivial vs the testing complexity. We're not Google — we don't need an experimentation layer.

### Comparison summary

|  | A (piecemeal) | B (big-bang, chosen) | C (opt-in mode) |
|---|---|---|---|
| Confidence-coloured live tail | ❌ blocked | ✅ shipped | ⚠️ off-by-default |
| Worker API change | None | Breaking once | Additive |
| Decode cost (live tick) | Same | +~5% | Toggleable |
| Persisted-data migration | Same eventually | One coordinated migration | Eventually needed |
| Testing burden | 3× releases | 1× release | 2× matrix forever |
| Time to first user-visible win | 1-2d (smallest feature first) | 5-7d (all three at once) | 3-4d (toggle + smallest feature) |

## Chosen approach: Option B — coordinated worker-API release

### High-level shape after change

```
                       Whisper worker (return_timestamps: "word")
                                       │
                                       │ { type: "result", chunks: Word[], id, durationMs }
                                       ▼
                       whisperClient.ts (returns WindowResult2)
                                       │
                                       ▼
              Agreement<Word>          ┐
              (LocalAgreement-2        │      → live tail renders Word[] with confidence-tint span
               over Word[] not         │      → committed words flow as Word[] into transcriptSegments
               string[])               ┘
                                       │
                                       ▼
              transcriptSegments: TranscriptSegment2[]
                  { words: Word[], tMs }      (v0.5 shape — backward-compat reader handles v1)
                                       │
                       ┌───────────────┼───────────────┐
                       ▼               ▼               ▼
                  .txt export     .vtt export     .srt export
                  (collapses to   (per-word       (per-word
                   words.text)    cues OR         cues OR
                                  sentence cues   sentence cues
                                  by toggle)      by toggle)
                                       │
                                       ▼
                  sessionStore (EXPORT_VERSION=2)
                  + reader-side compat for v1 bundles
```

### `Word` shape

```ts
export interface Word {
  text: string;            // The word, including leading space if any (matches what transformers.js emits)
  tStartMs: number;        // Start of this word in the audio window, in ms from window start
  tEndMs: number;          // End of this word in the audio window, in ms from window start
  confidence?: number;     // [0..1] — only present if encoder confidence path is wired (see OQ #2)
}
```

Plain object. No methods, no class. Serializes cleanly to JSON for shareLink + sessionStore.

### Backward-compat for stored sessions

`StoredSession` gains an optional `version?: 1 | 2` discriminator. Reader:

```ts
function loadSegments(s: StoredSession): TranscriptSegment2[] {
  if (!s.version || s.version === 1) {
    // v0.4.x — words: string[]. Wrap into Word[] with synthetic uniform timing.
    return s.transcript.map((seg) => ({
      tMs: seg.tMs,
      words: (seg.words as string[]).map((w, i, arr) => ({
        text: w,
        tStartMs: 0,
        tEndMs: Math.round(1000 / arr.length),  // uniform fake spacing within segment
      })),
    }));
  }
  return s.transcript as TranscriptSegment2[];
}
```

Synthetic timing for v1 sessions means old transcripts STILL export to .vtt/.srt with reasonable-looking (if uniform) per-word cues. No data loss. No "version 1 sessions can't be exported" wall.

## Migration plan — 8 commits

Order is load-bearing. Each commit is independently revertible and shippable.

### Commit 1 — Worker emits `chunks: Word[]`

`public/whisper-worker.js` flips `return_timestamps: "word"` and the result handler builds a `chunks` array from the transformers.js output. Parent receives BOTH `text` (kept for backward-compat) AND `chunks`.

- File: `public/whisper-worker.js` only
- Shape: `{ type: "result", text, chunks: Word[], id, durationMs }`
- No downstream consumer reads `chunks` yet — purely additive
- Test: a 5s sample of known audio → verify `chunks` length matches word count, monotonic `tStartMs`

**Scope:** ~40 LOC in worker, no other files
**Risk:** Low — additive worker payload, downstream still uses `text`

### Commit 2 — Plumb `chunks` through `whisperClient.ts` to a new `transcribeWindow2()` method

Add a parallel `transcribeWindow2(audio): Promise<WindowResult2>` that returns the new richer shape. Keep `transcribeWindow()` returning `{ text, durationMs }` for now.

- Files: `src/lib/whisperClient.ts`, new exported `Word` type from `src/lib/transcript.ts`
- New consumers can opt into v2; old code path untouched
- Add 4 vitest cases — Word type round-trip, empty chunks → empty array, malformed chunks → safe fallback

**Scope:** ~80 LOC + tests
**Risk:** Low — purely additive API

### Commit 3 — `Agreement<Word>` (generic refactor)

`agreement.ts` currently operates on `string[]`. Generic over a "tokenizable item" type — agreement key is `.text.toLowerCase().trim()`, committed/live return `T[]` not `string[]`. Existing string-mode callers wrap with `text: w` to preserve old semantics.

- Files: `src/lib/agreement.ts`, `src/lib/agreement.test.ts`
- Existing 14 tests get updated to use the wrapped shape
- Add 6 new tests covering: identity agreement on Word inputs, confidence preserved through commits, stable-prefix detection unaffected by tStartMs differences

**Scope:** ~150 LOC + 6 new tests; ~20 LOC of test-file rewrites
**Risk:** Medium — agreement is the streaming hot path, regressions here = no live captions

### Commit 4 — Switch live caption loop to v2 path

`CaptionApp.script.ts` swaps `transcribeWindow` for `transcribeWindow2`, agreement for `Agreement<Word>`, and `appendCommittedWords(string[])` becomes `appendCommittedWords(Word[])`. Render path reads `.text` from each Word. **No new visible feature yet** — same captions, same UX, just running on richer data.

- File: `src/components/CaptionApp.script.ts`
- Existing silence guard + hallucination guard + paragraph-break logic unchanged (operates on `.text`)
- Visible: nothing changes for the user
- This is the load-bearing commit — if it regresses, EVERYTHING regresses

**Scope:** ~200 LOC of touch (mostly type signatures)
**Risk:** Medium-high — touches the streaming hot path; must manually QA the four core flows (tab capture, mic capture, sample, share-link viewer) before merge

### Commit 5 — Bump EXPORT_VERSION to 2 + v1 reader fallback

`sessionStore.ts` writes `version: 2` + `Word[]` shape. Reader recognizes both v1 and v2 and synthesizes word timestamps for v1 sessions (uniform spacing within segment). shareLink encoder/decoder follows the same dual-version path.

- Files: `src/lib/sessionStore.ts`, `src/lib/shareLink.ts`, plus tests for both
- Existing 22 sessionStore tests + 17 shareLink tests must keep passing
- Add ~10 new tests covering v1→v2 read path, mixed-version IndexedDB, v2 round-trip

**Scope:** ~120 LOC + ~10 tests
**Risk:** Medium — persisted data migration; once v0.5 ships, downgrading a user back to v0.4.x breaks their NEW sessions (but old ones still work for them)

### Commit 6 — Word-level .vtt / .srt with sentence-grouping toggle (Phase 2.5)

`transcript.ts` gains a second formatter signature. New prefs key `livecaptionit:export-granularity` with values `"segment"` (current) / `"sentence"` (group words into ~3-4s cues at punctuation) / `"word"` (one cue per word — debug). Default `"sentence"`. UI: 3-radio control in the download bar (next to .vtt / .srt buttons) appears only when transcript has v2 segments.

- Files: `src/lib/transcript.ts`, `src/components/CaptionApp.astro`, `src/components/CaptionApp.script.ts`
- Sentence grouping splits at `[.!?]` then enforces 3-4s max duration with mid-sentence break
- Add ~15 vitest cases covering all three granularities + boundary edge cases

**Scope:** ~250 LOC + ~15 tests
**Risk:** Low — purely on export path, doesn't touch live captioning

### Commit 7 — Confidence-coloured live tail (Phase 1.7)

Live tail span gets a per-word `<span class="conf-low|conf-mid|conf-high">` wrap based on `Word.confidence`. CSS in `global.css` paints `conf-low` at 50% opacity, `conf-mid` at 75%, `conf-high` at 100%. Threshold: `<0.5 → low, 0.5-0.8 → mid, >0.8 → high`. Confidence is OPT-IN via the same prefs panel (`livecaptionit:show-confidence`, default off) because users new to STT find the visual noise distracting.

- Files: `src/components/CaptionApp.script.ts`, `src/styles/global.css`, `src/components/CaptionApp.astro` (prefs toggle)
- If `Word.confidence` is undefined (encoder path not wired — see OQ #2), gracefully no-op (no conf-* class)
- Manual QA only — confidence rendering is visual, no unit test value-add

**Scope:** ~80 LOC
**Risk:** Low — additive visual layer over already-rendered words

### Commit 8 — Speaker-turn manual editing (Phase 3.6)

Session viewer dialog gains an Edit button. Edit mode replaces the rendered transcript with a contenteditable list of word `<span>`s. User can: click a word to enter inline-edit, delete words (Backspace on selected), insert words (right-arrow at word boundary → type), mark a paragraph break (Shift+Enter splits the current segment). Save updates the StoredSession in IndexedDB. Cancel reverts.

- Files: `src/components/CaptionApp.astro` (dialog markup), `src/components/CaptionApp.script.ts` (editor logic), new `src/lib/transcriptEditor.ts` (pure-function edit primitives)
- Per-word DOM with `data-word-idx` lets edits anchor to the original Word index → timing preserved on save
- 12-15 vitest cases for the pure-function primitives (insertWord, deleteWord, splitSegment, joinSegments)

**Scope:** ~350 LOC + 12-15 tests
**Risk:** Medium — contenteditable is famously hairy; mitigated by keeping mutation logic in a pure-function module that's exhaustively tested

## Non-goals

- **No multi-speaker labels.** Even with word timestamps we don't get speaker identity. The contenteditable editor lets users manually prepend `Alice: ` / `Bob: ` but the app provides no automation. Real diarization stays a v0.6 (pyannote port) story.
- **No mid-tick worker reconfiguration.** `return_timestamps: "word"` is set at worker init time and stays for the session. Users can't toggle it on/off mid-recording.
- **No re-decode for old (v1) stored sessions.** Synthetic uniform per-word timing for v1 reads is "good enough" — we don't re-run Whisper to give them real timestamps.
- **No streaming chunks during decode.** Worker still returns one batch per tick. Sub-tick "appearing word-by-word" UX is a v0.6 story (would require WebGPU stream-decode).
- **No transcript editor on the live caption box.** Edit happens AFTER stop, in the session viewer. Editing live captions would race the streaming agreement and create a UX nightmare.

## Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `return_timestamps: "word"` is slower than `false` for some model variants | Medium | Adds ~5-15% to decode time → may exceed tick budget on weak devices | Adaptive-tick logic already exists; would just back off naturally. Add a `useTimestamps: false` worker flag as escape hatch in case of catastrophic regression. |
| Word-timestamp output format from transformers.js changes shape between versions | Medium | Silent breakage of v0.5 worker | We pin `3.7.5` of transformers.js (see project skill). Add an explicit shape check on first worker `chunks` payload that logs+falls-back to text-only mode on shape mismatch. |
| Persisted-data migration race: user opens v0.5 in tab A while v0.4.x is still cached in tab B | Low | Tab B writes a v1 session AFTER v0.5 has read its store as "all v2" | sessionStore reads version per-session at retrieval time, not at store-init time. Already the case. Just verify in commit 5's test suite. |
| Contenteditable in commit 8 produces nested DOM from copy-paste | High | Edit save corrupts segment shape | Strip all formatting on paste via clipboard event handler; only allow plain text. Add explicit anti-corruption integration test. |
| `Agreement<Word>` accidentally compares `Word` objects by reference instead of `.text` | Medium | LocalAgreement-2 never finds stable prefixes → captions stall | Agreement key extractor must be passed in (`(w: Word) => w.text.toLowerCase().trim()`); not implicit `===`. Tested in commit 3. |
| Confidence values from transformers.js are normalized differently across models (whisper-base vs whisper-large-v3-turbo) | Medium | conf-low/mid/high thresholds wrong on large model | Compute thresholds from observed distribution per-session (5th / 50th / 95th percentile) instead of fixed cutoffs. Adaptive but interpretable. |

## Open questions

1. **Should we ship commits 1-4 (the plumbing) as v0.4.4 BEFORE v0.5, to derisk the visible features?**
   Pro: zero user-visible change → safe to ship → cuts v0.5 from 8 commits to 4 visible ones. Con: a "release" with no user-visible value is unusual for an indie tool, no marketing angle. **Recommend YES**, ship as v0.4.4 with the changelog framed as "groundwork for v0.5", then commits 5-8 land as v0.5 over the following days.

2. **Where do we get per-word confidence from?**
   transformers.js Whisper does NOT expose per-word logprob in the default `chunks` output. Two options:
   - (a) Compute from token-level logits via a worker patch (we'd need to monkey-patch the pipeline's generate fn — fragile, version-sensitive)
   - (b) Approximate from token-position variance across LocalAgreement-2 ticks (words that took many ticks to stabilize = low confidence). This is a heuristic but it lives entirely in OUR code, no transformers.js patching.
   **Recommend (b)** for v0.5 — call it "stability-based confidence" in code comments to be honest about what it measures. (a) is a v0.6 story if/when transformers.js gains a stable confidence API.

3. **Sentence-cue grouping in commit 6 — when no `.!?` punctuation appears for >10s, what do we do?**
   Whisper's punctuation is decent but not perfect on long monologues. Options: (i) force-break at 10s regardless, (ii) use word-rate heuristic (>250ms gap = sentence break), (iii) just emit one 10s+ cue. **Recommend (i)** — 10s is already past the comfortable subtitle limit; users would rather have an arbitrary break than a wall-of-text cue.

4. **Should commit 8's editor support undo/redo?**
   Real editors do, contenteditable's native undo is unreliable. Building our own undo stack adds ~150 LOC. **Recommend NO for v0.5** — ship single-step "Cancel" (reverts whole edit session). Add undo in a v0.5.1 if users ask.

5. **Domain naming for the bigger story — should we be called "LiveCaptionIt" or "LiveCaption.app" / "Captions.live" given how far we've gone past "just live captions"?**
   Out of scope for this doc but worth noting — by the time word-level edit + per-word confidence + sentence cues ship, the app is closer to an in-browser transcription product than a caption widget. Surface to Shrestha after v0.5 lands.

## Success criteria

1. **Confidence-coloured live tail toggleable from prefs:** with confidence ON, words appear at 50% opacity then strengthen to 100% as agreement firms them up. With confidence OFF, behavior is byte-identical to v0.4.3.
2. **Export a session to .vtt with `Granularity: Sentence` selected:** every cue is ≤4s long and breaks at punctuation when available. Open the .vtt in VLC over a video — subtitles feel natural, not segment-clumped.
3. **Restore a v0.4.x JSON export in v0.5:** the imported session appears in Recent sessions, opens in the viewer, and exports successfully to .vtt/.srt (with synthetic uniform per-word timing — acceptable).
4. **Open a v0.4.x recorded session in v0.5's editor:** the editor renders fine, edits save (with synthetic timing preserved), reopening the edited session shows the changes.
5. **Edit a v0.5-recorded session:** click a misheard word, replace it, save. Re-export to .srt. Edited word appears in the exported file with the correct (real) timestamp from the original recording.
6. **Stress test:** 10-minute capture with whisper-base. Live tail confidence rendering does not noticeably impact frame rate. Edit-mode opening on the 10min transcript renders within 300ms (~1500 words).

When all hold, ship as v0.5.0.

## Appendix: why not just use OpenAI Whisper API for the editor's "fix this segment" feature?

Tempting: rather than building a contenteditable editor, add a "Send this segment to GPT-4 for cleanup" button. Rejected because:
- Defeats the "audio never leaves device" core value prop (see `site.config.ts` brand line + privacy page)
- Recurring cost for users (or for us if we eat it) → kills the free-forever pricing
- Network dependency on a tool whose whole point is "works offline / local-only"

If we ever want LLM-cleanup, it'd be via a local WebGPU LLM (web-llm + Phi-3 / Qwen2-0.5B) running in the same worker pool. v0.6+ story, not v0.5.

## Appendix: why not WebGPU stream-decode for sub-tick word appearance?

WebGPU stream-decode (where Whisper emits words as they're decoded, not waiting for the full window to finish) would give the most magical UX — words appearing letter-by-letter as the model thinks. Rejected for v0.5 because:
- transformers.js doesn't support it yet (would need to fork the pipeline)
- Adds another major API surface to migrate
- The LocalAgreement-2 commit pattern handles "appearing" perceptually well already

If transformers.js gains streaming support natively in 4.x, revisit for v0.6.
