## Feature 1 — Gap-based turn detection

### Why
Right now committed words flow into ~14-word paragraphs purely by word count. A 5-minute meeting transcript ends up as ~20 indistinguishable paragraphs. With gap-based breaks, every time the audio goes silent for ≥1.5s (someone finishes a sentence, the other person starts), we start a fresh paragraph WITH a subtle visual separator. Reads like meeting notes instead of a wall of text.

### Honest framing
Inline FAQ + about-page copy will call this **"turn-based paragraph breaks"** — NOT "speaker diarization". The latter implies "Speaker A vs Speaker B" which we cannot deliver from mono audio. The about page explicitly clarifies the limitation.

### What changes
- `audioCapture.ts`: already tracks `lastNonSilentMs` via the `onLevel` callback (RMS-based). Expose this to script.ts via a new optional capture option `onSilenceTransition: (silentMs: number) => void` fired ONCE when audio crosses from silent → speaking again, with the duration of the silent gap.
- Actually simpler: keep silence tracking inside script.ts (it already lives there for the silence guard), add a new state machine on top of it: when transitioning from "silent for ≥1.5s" → "speaking again", mark `forceParagraphBreak = true`. Next `appendCommittedWords()` call starts a new `<p>` instead of appending to the last one.
- The paragraph break visual = a thin horizontal rule via CSS (`hr-turn` class, 1px line at `--color-border` opacity 0.5). Subtle, not screamy.
- Constant: `TURN_GAP_SECONDS = 1.5` in script.ts (NOT in audioCapture, scoped to UI logic).

### What does NOT change
- The silence guard (2.5s reset of rolling buffer + agreement) stays exactly as is. Turn detection runs on a SHORTER timer with DIFFERENT semantics (visual break vs state reset) — no overlap.
- The `MAX_LINE_WORDS = 14` paragraph cap still applies as a fallback (long monologue without gaps still gets broken up).
- Transcript download formats (.txt/.vtt/.srt): paragraph breaks become double-newlines in .txt. .vtt and .srt stay segment-based since they're segment-level by spec.

### Verification
- All existing 92 vitest pass.
- ~3 new unit tests for the turn-break logic if we extract it to a pure function (likely YES — `shouldStartNewParagraph(silenceMs, threshold)` is trivially testable).
- Manual smoke:
  1. Tab capture of a 2-speaker podcast → visible paragraph breaks at speaker swaps
  2. Tab capture of a monologue (continuous speech) → falls back to 14-word breaks, no turn lines
  3. Silence guard still fires at 2.5s (no regression in hallucination prevention)
  4. PiP shows turn breaks too (they're DOM, not styling overrides)
- About + FAQ updated with "turn-based breaks (not speaker identity)" disclosure.

## Feature 2 — Session search

### Why
Once history hits 10+ sessions, the "Recent sessions" list on idle gets cluttered. A search box that filters by transcript content (or session source/date) makes the history actually useful for "what did Sarah say about the Q3 plan?"

### What changes
- New `src/lib/sessionSearch.ts` — pure function `searchSessions(sessions, query)` returns matching subset with simple case-insensitive substring match across the joined transcript text + sourceLabel + ISO date string. Returns sorted by recency (preserved order from input).
- `CaptionApp.astro`: search input above the Recent sessions list (only renders when ≥3 sessions exist — under that, search is friction).
- `CaptionApp.script.ts`: wire input → re-render history list with filter, debounce 150ms.
- Empty-state when query has no matches: "No sessions match — try another search."
- Also surfaces in the history-viewer dialog if we add a "see all" link to it later (out of scope for v0.4.2).
- 8-10 vitest cases for `searchSessions`: empty query returns all, single word matches, multi-word AND, case-insensitive, no-match returns [], source/date filters work.

### Verification
- Manual smoke: search a known word → only matching sessions shown. Clear input → all sessions return. Search rare phrase → empty state.
- All vitest pass (92 existing + ~9 new = ~101).

## Feature 3 — Session export/import as JSON

### Why
Three motivations: (a) backup story for users with valuable transcripts, (b) cross-browser portability ("I started session in Chrome, want to view in Firefox"), (c) future "share a transcript" features depend on a stable export format existing.

### What changes
- `src/lib/sessionStore.ts`: add `exportAllSessions(): Promise<ExportBundle>` returning `{ version: 1, exportedAt: ISO, sessions: StoredSession[] }`. Add `importSessions(bundle): Promise<{ imported: number, skipped: number }>` that inserts each session via `addSession` skipping duplicates by ID. Validates `version: 1` strictly; rejects unknown versions with clear error.
- `ExportBundle` shape locked in v1: future versions get a migration path, not a silent break.
- `CaptionApp.astro`: in the history viewer dialog (which already shows full transcript), add two new buttons next to Clear all:
  - **Export all** — downloads `livecaptionit-history-YYYY-MM-DD.json`
  - **Import** — opens file picker → reads JSON → confirms count + asks before merge
- `CaptionApp.script.ts`: wire both buttons. Use existing `downloadString()` from transcript.ts for the export download (same pattern as .txt/.vtt/.srt).
- Import flow respects the 20-session cap — if importing 15 sessions into a store with 18, oldest get pruned to keep newest 20.
- 6-8 vitest cases for export/import roundtrip: empty store exports valid bundle, full roundtrip preserves all fields, unknown version rejects, duplicate IDs skip, 20-cap enforced.

### Verification
- Manual smoke:
  1. Run 3+ sessions, click Export all → JSON downloads
  2. Clear all → empty list
  3. Click Import → pick the JSON → 3 sessions reappear with original timestamps
  4. Try to import a malformed JSON → error message, history unchanged
- Privacy page mention: "exported transcripts contain the same text shown in your browser — no audio, no metadata beyond what's listed in the file"
