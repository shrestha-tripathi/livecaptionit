## Feature 1 — AudioWorklet migration

### Why
`ScriptProcessorNode` is deprecated. Runs on main thread → jank under
load (worker is already CPU-heavy). AudioWorklet runs on audio thread,
real-time priority, lower latency, no main-thread blocking. v0.1 deferred
it for simplicity; v0.4.1 is the right moment because (a) the rest of
the pipeline is stable and (b) we're about to add pause/resume which
benefits from the cleaner suspend semantics.

### What changes
- New file `public/audio-worklet-capture.js` — standalone AudioWorklet
  processor module (cannot use ES imports; pure JS class).
- `wireAudioPipeline` in `src/lib/audioCapture.ts`:
  - Replace `ctx.createScriptProcessor(4096, 1, 1)` + `onaudioprocess`
    with `await ctx.audioWorklet.addModule(...)` +
    `new AudioWorkletNode(ctx, "capture-processor")` +
    `node.port.onmessage`.
  - Accumulation: worklet's `process()` is called every 128 samples
    (~8ms). Posting per-frame = ~125 msg/sec = wasteful. Worklet
    accumulates internally to 4096-sample frames before posting (matches
    current `PROCESSOR_BUF`). Yields ~37 msg/sec — same downstream
    behavior, lower message overhead.
  - Manual 16kHz resample stays in main-thread message handler (Whisper
    pipeline already expects 16kHz; worklet operates at ctx rate).
    *Decision: keep resample on main thread for now, can push it into
    worklet later if perf demands.*
  - Transferable `Float32Array.buffer` on every post (zero-copy).
  - Worklet must be loaded async → `wireAudioPipeline` becomes async.
    Both `startCapture` and `startMicCapture` already use `await` so
    this propagates cleanly.

### Worklet file shape (sketch)
```js
// public/audio-worklet-capture.js
class CaptureProcessor extends AudioWorkletProcessor {
  constructor() {
    super();
    this._buf = new Float32Array(4096);
    this._pos = 0;
  }
  process(inputs) {
    const input = inputs[0]?.[0];           // first input, first channel
    if (!input || input.length === 0) return true;
    let i = 0;
    while (i < input.length) {
      const remaining = 4096 - this._pos;
      const take = Math.min(remaining, input.length - i);
      this._buf.set(input.subarray(i, i + take), this._pos);
      this._pos += take;
      i += take;
      if (this._pos === 4096) {
        // Allocate fresh + transfer so we don't ship a buffer the
        // audio thread will overwrite next frame.
        const out = new Float32Array(this._buf);
        this.port.postMessage(out, [out.buffer]);
        this._pos = 0;
      }
    }
    return true;  // keep processor alive
  }
}
registerProcessor("capture-processor", CaptureProcessor);
```

  lacks WebGPU / Document PiP and can't use the app anyway.

  AudioContext (rare), the linear downsample must still work on the
  worklet-accumulated frame. Same code path as today, just moved.

## Feature 2 — Space pause/resume

### Why
Deferred from v0.4.0. Highest-leverage missing shortcut: live captioning
a meeting where someone interrupts you — you want to pause without
losing your buffer or re-prompting the picker. `Esc` is "stop entirely",
`Space` is "freeze and come back".

### Default decisions (push back if you disagree)
1. **Pause behavior** = `AudioContext.suspend()` + (for sample mode) also
   pause the HTMLAudioElement. When suspended:
   - Worklet's `process()` stops being called → no new samples flow
   - Tick scheduler skips ticks while paused (no point re-transcribing
     identical buffer; would waste GPU)
   - Rolling buffer freezes at current length
   - Committed transcript stays visible; live tail freezes
2. **Resume behavior** = `AudioContext.resume()` + (sample) `audio.play()`.
   Tick scheduler resumes naturally.
3. **UI for paused state** = new substate `paused` on the active panel
   (extends the existing `data-substate="live|stopped"` pattern):
   - LIVE pill swaps to a neutral "PAUSED" pill (no animated dot)
   - "Stop" button stays visible (you can still stop while paused)
   - Subtle hint replaces the live shortcut hint: `Space resume · Esc stop`
   - "Pop out" button stays visible if not in PiP
   - **No dedicated "Resume" button** — Space is the affordance, hint
     advertises it. If user feedback says invisible, add a button next
     ship.
4. **Persistence** = pause state is in-session only. Stop + Start new
   resets everything.
5. **Worklet bundling** = separate `public/audio-worklet-capture.js`
   file (not Blob URL inline). Easier to debug, served by CF Pages with
   correct MIME, browser can cache it.

### What changes
- `src/lib/audioCapture.ts`:
  - Extend `CaptureHandle` interface with `pause(): Promise<void>` and
    `resume(): Promise<void>`. (`Promise` because `ctx.suspend/resume`
    return promises.)
  - Add `isPaused` getter (not stored on handle — read from `ctx.state`).
- `src/lib/sampleFeed.ts`:
  - `SampleCaptureHandle` (which already mirrors `CaptureHandle`) gains
    matching `pause`/`resume`. Implementation pauses the HTMLAudioElement
    too so user hears playback stop.
- `src/lib/shortcuts.ts`:
  - Add new `ShortcutContext` value: `"active-paused"`.
  - Existing `"active-live"` shortcuts (Esc, P) also valid in
    `"active-paused"` (add to `contexts` array).
  - New `Space` binding: `contexts: ["active-live", "active-paused"]`,
    section `"Capture"`, label `"Pause / resume capture"`.
- `src/components/CaptionApp.script.ts`:
  - New state tracker: `let isPaused = false`.
  - Update `getShortcutContext()` mapper: return `"active-paused"` when
    `isPaused`, else `"active-live"`.
  - Tick scheduler: early-return `if (isPaused) { scheduleNextTick(); return; }`
    in the tick loop (still re-arm the timer so it picks up on resume).
  - New `togglePause()` handler called by Space shortcut: flips
    `isPaused`, calls `captureHandle.pause()/resume()`,
    calls `setSubstate("paused"|"live")`, updates status banner text
    (`"Paused"` vs `"Listening…"`).
  - In `stopPipeline()`: reset `isPaused = false`.
- `src/components/CaptionApp.astro`:
  - Add `data-when="paused"` markup: new PAUSED pill, paused hint text.
  - Update help dialog table to include the Space row.
  - Update footer shortcut hint: `Esc stop · Space pause · P popout · ? help`.
- `src/styles/global.css`:
  - `.cp-pill-paused` style (gray dot, gray text, no animation).
- `src/lib/shortcuts.test.ts`:
  - 4-5 new cases: Space matches in active-live, Space matches in
    active-paused, Space does NOT match in idle/loading/active-stopped,
    other shortcuts still work in active-paused.

### Verification
- All vitest pass (existing 87 + ~5 new = ~92).
- Manual smoke:
  1. Start tab capture → talk → Space → confirm "PAUSED" pill, captions
     freeze, no GPU activity, no new committed words
  2. Space again → "LIVE" pill returns, captions resume
  3. Start mic → same flow
  4. Start sample → Space → audio actually stops playing
  5. Pop out → Space (from PiP doc) → pause works in PiP too
  6. Stop while paused → enters stopped substate correctly
  7. Esc while paused → stops correctly
  8. Help dialog shows Space row, footer hint includes Space

### Risk
- **PiP focus:** Space inside PiP doc must fire the shortcut (PiP doc
  events don't bubble to parent). Existing `setPipShortcuts()` already
  installs shortcuts on both docs, so this works automatically.
- **Browser typing-context exception:** `isTypingContext()` already
  guards inputs/textareas. Space in `<button>` focus context: the
  shortcut module always calls `preventDefault()`, so the focused
  button's default Space behavior (click) is suppressed. **Decision:**
  this is acceptable; user has to click physically or Enter for buttons.
  Add a defensive Tab focus → first-pause-attempt sanity check in QA.
- **Resume mid-tick:** if user pauses mid-transcription (worker call
  in flight), the result still arrives after resume. Existing
  `inFlight` guard + agreement re-converges naturally; no special
  handling needed.
